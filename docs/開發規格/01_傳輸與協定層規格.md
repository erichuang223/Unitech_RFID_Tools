> CHECKSUM 計算、封包編碼/解碼與回包路由。架構定位見 [00 總覽與架構](00_總覽與架構.md)。

## 1. 傳輸層（Transport Layer）

### 1.1 職責

- 列舉、開啟、關閉序列埠（COM port）。
- 設定通訊參數。
- 以位元組串流收發資料。
- 將連續的 byte stream **切分成一筆筆完整封包**（分包 / framing）後往上層交付。

### 1.2 介面（建議）

```csharp
public interface ISerialTransport : IDisposable
{
    IReadOnlyList<string> GetAvailablePorts();      // SerialPort.GetPortNames()
    void Open(string portName, int baudRate);       // 預設 8/N/1
    void Close();
    bool IsOpen { get; }

    void SendRaw(byte[] bytes);                      // 直接送出位元組（已組好的封包）
    event EventHandler<byte[]> RawReceived;          // 收到的原始位元組（給 hex 監看）
    event EventHandler<Packet> PacketReceived;       // 分包完成的封包（給協定層）
}
```

### 1.3 通訊參數

| 參數 | 預設 | 說明 |
|---|---|---|
| BaudRate | 115200 | 可選 115200 / 460800 / 921600（見 [5.45/5.46](../Chapter5c_RFIDConfiguration_3.md)）；實際預設值以 RM300P 硬體手冊為準 |
| DataBits | 8 | |
| Parity | None | |
| StopBits | 1 | |
| Handshake | None | 硬體有 CTS/RTS 腳位，第一版預設不用流控；若實測需要再開 |

### 1.4 收發與執行緒模型

- 讀取在**背景執行緒/非同步**進行（`SerialPort.DataReceived` 或 `BaseStream.ReadAsync`），
  避免阻塞 UI 光棒選單。
- 收到位元組先進**接收緩衝區（ring/累積 buffer）**，再交由分包器處理。
- 分包出的封包與原始位元組分別以事件 `PacketReceived` / `RawReceived` 往上拋；
  UI 與協定層各自訂閱（盤點串流、hex 監看都靠事件驅動）。

### 1.5 分包演算法（Framing）★關鍵

封包結構：`STX(0x02) | LENGTH(2) | ID(2) | [STATUS(1)] | [DATA(n)] | CHECKSUM(2) | ETX(0x03)`

> **不可用「掃描到 0x03 就當結尾」的天真做法**——因為 DATA / CHECKSUM 內可能剛好出現
> `0x02` 或 `0x03`。必須**以 LENGTH 欄位決定封包長度**。

**LENGTH 的定義（已用原文範例驗證）**：
`LENGTH = 「LENGTH 欄位本身(2) + ID(2) + STATUS(若有) + DATA」的位元組數`。
亦即整包總長 = `1(STX) + LENGTH + 2(CHECKSUM) + 1(ETX)` = `LENGTH + 4`。

分包步驟：

1. 在緩衝區中往前找到 `STX = 0x02`，丟棄它之前的雜訊位元組。
2. 確認 STX 之後至少還有 2 個位元組，讀出 **LENGTH（2 byte，大端序 big-endian）= L**。
3. 一個完整封包在 STX 之後需要 `L + 3` 個位元組（= `(L-2) 的 ID/STATUS/DATA + 2 CHECKSUM + 1 ETX`，
   再加上已含在 L 內的 2 個 LENGTH 位元組）。若緩衝區尚不足，**等待更多資料**。
4. 取出 `L` 個位元組（LENGTH 欄位起算到 DATA 結束）作為 **checksum 輸入區**。
5. 讀出其後 2 個位元組為 **CHECKSUM（大端序）**，再 1 個位元組應為 **ETX = 0x03**。
6. 驗證：ETX 正確、CHECKSUM 正確（見 §2.2）。通過 → 組成 `Packet` 上拋；
   失敗 → 記錄錯誤、丟棄此 STX 並從下一個位元組重新尋找 STX。

> 多位元組欄位（LENGTH、ID、CHECKSUM）一律**大端序（高位元組在前）**，已由原文範例驗證。

### 1.6 逾時與重連

- **指令逾時**：送出同步指令後若於逾時時間（建議預設 1000 ms，可設）內未收到對應回包，回報逾時錯誤。
- **重連**：偵測到埠中斷（拔線/例外）時，標記斷線並通知 UI；提供手動重連。

## 2. 協定層（Protocol Layer）

### 2.1 封包模型

```csharp
public sealed class Packet
{
    public ushort Id { get; init; }          // 2-byte 指令/回包 ID（大端序解出）
    public byte[] Payload { get; init; }      // STATUS + DATA（回應）或 DATA（資料送出）；不含 ID
    public byte[] Raw { get; init; }          // 完整原始位元組（含 STX..ETX），供 hex 監看
}
```

> 是否把 STATUS 從 Payload 拆出，由解析器依封包類型決定（見 §2.4）。

### 2.2 CHECKSUM 演算法 ★關鍵（已驗證）

**定義**：對「checksum 輸入區」的每個位元組做**逐位元組相加**，取其 **16-bit 二補數（two's complement）**，
以**大端序 2 byte** 輸出。

- **輸入區** = `LENGTH(2) + ID(2) + STATUS(若有) + DATA`，即整包扣掉 STX、CHECKSUM、ETX 後的位元組
  （等同分包步驟取出的那 `L` 個位元組）。
- 公式：`CHECKSUM = (0x10000 - (Σ bytes & 0xFFFF)) & 0xFFFF`

```csharp
public static ushort CalcChecksum(ReadOnlySpan<byte> lengthToDataInclusive)
{
    int sum = 0;
    foreach (byte b in lengthToDataInclusive) sum += b;
    return (ushort)((0x10000 - (sum & 0xFFFF)) & 0xFFFF);
}
```

**驗算範例**（取自原文）：

| 來源 | 輸入區位元組 | 位元組和 | 二補數 = CHECKSUM | 原文封包 |
|---|---|---|---|---|
| [2.1](../Chapter2_GeneralCommands.md) Host 讀版本 | `00 04 00 21` | 0x25 | **0xFFDB** | `…<0xFF 0xDB>…` ✓ |
| [2.1](../Chapter2_GeneralCommands.md) RFID 回版本 | `00 09 00 21 00 00 00 00 01` | 0x2B | **0xFFD5** | `…<0xFF 0xD5>…` ✓ |
| [4.1](../Chapter4_RFIDDataReply.md) Inventory Data | `00 15 00 C0 01 E9 62 34 00 AA AA 34 12 DC 03 01 17 16 10 B9 88` | 0x64D | **0xF9B3** | `…<0xF9 0xB3>…` ✓ |

> 收包驗證：重算輸入區 checksum 與封包帶的 CHECKSUM 比對；不符即視為損毀封包。

### 2.3 封包編碼（送出 / Host Command）

依 [第 1 章 1.1](../Chapter1_CommandFormat.md)，主機指令有「無 DATA」與「含 DATA」兩式：

```
<STX><LENGTH><ID><CHECKSUM><ETX>
<STX><LENGTH><ID><DATA><CHECKSUM><ETX>
```

編碼步驟：
1. 組出 `ID(2, 大端) + DATA`。
2. `L = 2 + (ID+DATA 的長度)`；組出 `LENGTH(2, 大端)`。
3. 對 `LENGTH + ID + DATA` 算 CHECKSUM（§2.2）。
4. 串接 `STX + LENGTH + ID + DATA + CHECKSUM + ETX` 送出。

### 2.4 封包解碼與回包路由

收到的封包依 **ID** 分三類路由（ID 對照見各章）：

| 類別 | 判斷 | 處理 | 範例 ID |
|---|---|---|---|
| **同步回應**（Response） | ID 與剛送出的指令相同 | 取 `Payload[0]` 為 **STATUS**，其餘為 DATA；配對到等待中的指令請求 | 0x21、0x103、0x117 … |
| **非同步資料**（Data Send） | 盤點/存取結果類 ID | 解析為標籤資料/存取結果，以**事件**往上拋（不配對請求） | 0xC0/0xC1/0xC2、0xB0~0xB2、0xA0、0xC7、0xC8 |
| **錯誤 / 事件**（主動回報） | 錯誤或事件 ID | 解析後以事件往上拋 | 0xD0/0xD1（[4.10/4.11](../Chapter4_RFIDDataReply.md)）、0xE0/0xE1（[第 7 章](../Chapter7_EventNotification.md)） |

> **同步指令的配對機制**：送出指令後，等待「同一 ID」的回應或逾時。盤點 0x80 比較特別——
> 它的「同步回應」只是受理 STATUS，真正的標籤資料是後續**非同步**的 0xC0/0xC2 串流，需用事件處理。
> 這個「受理回應 vs 串流資料」的分流是盤點功能正確運作的關鍵，詳見 [02 指令層](02_指令層規格.md) 與 [03 UI](03_ConsoleUI與選單規格.md)。

### 2.5 STATUS 與錯誤碼

- STATUS 對照表（0x00 成功、0x01 忙碌、0x02 參數無效…）見 [第 1 章](../Chapter1_CommandFormat.md)。
- 非同步錯誤回包 OPERRCODE / SYSERRCODE 見 [4.10/4.11](../Chapter4_RFIDDataReply.md)；
  標籤存取錯誤 TAERRCODE 見 [4.9](../Chapter4_RFIDDataReply.md)。
- 協定層應把這些碼**解析成具名列舉 + 中文說明**，供 UI 直接顯示，不讓使用者自己查表。

### 2.6 可離線測試的部分

CHECKSUM 計算、封包編碼、分包器（餵入位元組陣列）、回包解析皆**不需要實體設備**，
應以單元測試覆蓋（用本篇與各章的範例封包當測試向量）。詳見 [04 確認測試](04_確認測試規格.md) 階段 -1。
