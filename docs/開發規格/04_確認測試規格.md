# 開發規格書 04 — 確認測試規格

> bench tool 開發規格書第 04 篇。定義「拿到設備後」用本工具進行的確認測試矩陣，
> 由低風險往高風險推進。每項測試列出目的、前置、步驟、預期、PASS/FAIL 判準。
> 指令對應見 [02 指令層](02_指令層規格.md)。

---

## 0. 測試總則

- **風險遞增原則**：先離線驗協定，再唯讀，再設定往返，最後才碰會改變標籤/設定的操作。
- **每項測試都要看兩件事**：①工具解析的中文結果是否正確；②原始 hex 收發是否符合預期封包。
- **實機行為以實測為準**：若實機與規格描述不同，**如實記錄**於測試結果，這是本工具的核心價值。
- 測試應準備：RM300P 設備、天線、若干**測試標籤**（可被覆寫，勿用正式資產標籤）。

---

## 階段 -1：離線協定驗證（不需設備）★最先做

| 項 | 目的 | 步驟 | PASS 判準 |
|---|---|---|---|
| -1.1 | CHECKSUM 算法正確 | 用 [01 §2.2](01_傳輸與協定層規格.md) 的三組範例向量算 checksum | 三組結果分別 = 0xFFDB / 0xFFD5 / 0xF9B3 |
| -1.2 | 封包編碼正確 | 以 API 組「讀版本 0x21」封包 | 位元組 = `02 00 04 00 21 FF DB 03` |
| -1.3 | 分包器正確 | 餵入連黏/分段/夾雜雜訊的位元組流，含 DATA 內帶 0x02/0x03 的封包 | 能正確切出每一包、不被假 ETX 騙到 |
| -1.4 | 回包解析正確 | 餵入各章範例回包（0xC0、0xC8、0xD0…） | 解析欄位與章節範例一致 |

> 階段 -1 全以單元測試實現，是後續所有測試的地基。

## 階段 0：連線

| 項 | 目的 | 步驟 | PASS 判準 |
|---|---|---|---|
| 0.1 | 列舉埠 | 開「連線 > 選擇 COM Port」 | 列出系統實際存在的 COM 埠 |
| 0.2 | 開埠 | 選埠 + baud → 連線 | 連線成功，標題列顯示●已連線 |
| 0.3 | 斷線/重連 | 拔線或斷線再重連 | 能偵測斷線並成功重連 |

## 階段 1：握手（協定試金石）★關鍵

| 項 | 目的 | 步驟 | PASS 判準 |
|---|---|---|---|
| 1.1 | 設備活著且協定正確 | 連線後執行「裝置資訊 > 韌體版本」(0x21) | 收到回包、ETX 正確、**CHECKSUM 驗證通過**、ID 對應、解出版本字串 |

> 1.1 通過 = 同時證明「設備活著」+「CHECKSUM 對」+「分包/解析對」。此關過了後面都是堆指令。
> 若失敗：先查 baud（換 460800/921600 試）、接線、CHECKSUM 實作。

## 階段 2：唯讀資訊巡檢（零風險）

| 項 | 步驟 | PASS 判準 |
|---|---|---|
| 2.1 | 讀機型 (0x22) | 得到合理機型字串（如 RM300P） |
| 2.2 | 讀序號 (0x24) / SKU (0x25) | 正確回應、可解析 |
| 2.3 | 讀環境溫度 (0x28) / PA 溫度 (0x29) | 溫度值落在合理範圍（換算 0.1°C） |

## 階段 3：設定 Get/Set 往返（低風險）

> **前置（重要）**：先把「設定儲存模式」設為 **RAM-only**（0x1F1，見 [05 安全](05_安全非功能與驗收.md)），
> 避免測試把設定寫進 Flash。每項採 **Get→Set→Get 驗證→還原** 流程。

| 項 | 目標設定 | 步驟 | PASS 判準 |
|---|---|---|---|
| 3.1 | 區域 Region (0x100/0x101) | Get 原值 → Set 新值 → Get 確認 → 還原 | 第二次 Get = 設定值；還原後 = 原值 |
| 3.2 | 天線功率 (0x102/0x103) | 同上，改某天線功率 | 讀回功率 = 設定值 |
| 3.3 | RF Mode (0x106/0x107) | 同上 | 讀回 = 設定值 |
| 3.4 | 盤點模式 (0x116/0x117) | 切連續/非連續 | 讀回模式 = 設定值（供階段 4 使用） |
| 3.5 | 失敗路徑 | 故意送無效參數 | 回 STATUS=0x02（參數無效），工具顯示中文錯誤 |

## 階段 4：盤點與讀寫（核心，中風險）★工具主目標

### 4A 盤點觸發三模式（對應使用者核心需求）

| 項 | 模式 | 步驟 | 預期（待實機驗證） | PASS 判準 |
|---|---|---|---|---|
| 4.1 | **單一觸發收集** | 設非連續 → 單次盤點收集 | 盤一輪 → 串流數筆 → 收到 `InventoryDone`(0xC7) → 自動停 | 即時表格出現標籤、收到 Done、自動停止 |
| 4.2 | **開啟連續收集** | 設連續 → 開啟連續收集 | 標籤持續刷新、不自動停 | 表格持續更新；計數持續增加 |
| 4.3 | **關閉連續收集** | 連續中按 Esc / 停止 | 送 Cancel(0x81) → 串流停止 | 停止後不再有新標籤；回受理 STATUS |
| 4.4 | 實機行為對照 | 比較 4.1~4.3 實測 vs 規格描述 | — | **如實記錄實機真實行為**（含與規格不同之處） |

> 4.4 是這份工具最重要的產出之一：用實測回答「設備到底怎麼觸發收集」，供應用端決定工廠現場收集策略。

### 4B 讀寫（限測試標籤）

| 項 | 步驟 | PASS 判準 |
|---|---|---|
| 4.5 | 讀標籤 (0x82) EPC/TID/USER | `TagAccessReported`(0xC8) 回正確資料，TAERRCODE=成功 |
| 4.6 | 寫標籤 (0x83) USER 區（**測試標籤**，二次確認） | 寫入後再讀回一致 |
| 4.7 | 無標籤情境 | 移開標籤再讀 | 回 TAERRCODE=No-Tag found（中文呈現） |

## 階段 5：例外與錯誤處理（不含破壞性操作）

> 範圍內**不做** Kill/Lock/Permalock；本階段聚焦「錯誤碼能否正確解析呈現」。

| 項 | 步驟 | PASS 判準 |
|---|---|---|
| 5.1 | 觸發操作錯誤 (0xD0) | 例如停用所有天線後盤點 | `ErrorReported` 顯示 OPERRCODE 中文（如「未啟用任何天線」） |
| 5.2 | 觸發系統錯誤 (0xD1) | 設極低溫度保護門檻誘發（謹慎） | 顯示 SYSERRCODE 中文（溫度過高） |
| 5.3 | 密碼錯誤 | 用錯誤 ACCPW 讀受保護標籤 | TAERRCODE=Invalid Password（中文） |

## 階段 6：效能與穩定性

| 項 | 步驟 | PASS 判準 |
|---|---|---|
| 6.1 | 連續盤點吞吐 | 多顆標籤連續盤點，觀察讀取速率 | 工具不掉包、不卡頓；可呈現約略 tags/秒（規格上限 >600 tags/s） |
| 6.2 | 長時間穩定 | 連續收集數十分鐘 | 無記憶體暴增、無崩潰、計數正常 |
| 6.3 | 多天線輪詢 | 啟用多天線、設盤點輪數 | 各天線皆有讀取，ANTID 正確標示 |

## 測試結果記錄

- 每次完整測試後，於本階段（階段 4/5/驗收）產生一份結果記錄（PASS/FAIL + 實機行為觀察 + 異常 hex）。
- 階段 4.4 的「實機觸發行為結論」應回寫到專案文件（例如階段 2 清單或新的測試報告），作為應用端決策依據。
