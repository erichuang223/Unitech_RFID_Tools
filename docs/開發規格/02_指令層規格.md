> bench tool 開發規格書第 02 篇。定義把 RM300P 指令集（[第 2~7 章](../Chapter2_GeneralCommands.md)）

## 1. 指令 API 通用模型

每個指令對應一個方法，遵循下列模式：

```csharp
public interface IRm300pClient
{
    // 同步指令：送出後等待同 ID 回應或逾時，回傳解析後結果（含 STATUS）
    Task<CommandResult<T>> SendAsync<T>(...);

    // 非同步串流事件（盤點 / 事件 / 錯誤）
    event EventHandler<TagReport>      TagReported;       // 0xC0/0xC1/0xC2/0xB0~0xB2/0xA0
    event EventHandler<InventoryDone>  InventoryDone;     // 0xC7
    event EventHandler<TagAccessData>  TagAccessReported; // 0xC8（讀取結果）
    event EventHandler<DeviceError>    ErrorReported;     // 0xD0/0xD1
    event EventHandler<DeviceEvent>    EventReported;     // 0xE0/0xE1
}

public sealed class CommandResult<T>
{
    public StatusCode Status { get; init; }   // 0x00=成功…（見第 1 章）
    public bool IsSuccess => Status == StatusCode.Success;
    public T? Data { get; init; }             // 回應 DATA 解析後的強型別資料
    public Packet Raw { get; init; }          // 原始封包（hex 監看）
}
```

**通則**：
- 同步指令一律回 `CommandResult<T>`，UI 直接拿 `IsSuccess` 與 `Data` 顯示。
- 非同步資料（盤點串流、事件、錯誤）一律走**事件**；呼叫端（UI）訂閱。
- 每個結果都保留 `Raw`，讓 hex 監看能對照。

## 3. 非同步資料解析（事件對應）

| 事件 | 觸發回包 ID | 解析為 | 章節 |
|---|---|---|---|
| `TagReported` | 0xC0/0xC1/0xC2、0xB0/0xB1/0xB2、0xA0 | `TagReport`（ANTID/RSSI/PC/EPC，視格式含 TID/相位/頻率/自訂） | [4.1~4.7](../Chapter4_RFIDDataReply.md) |
| `InventoryDone` | 0xC7 | 本輪盤點結束 | [4.8](../Chapter4_RFIDDataReply.md) |
| `TagAccessReported` | 0xC8 | `TagAccessData`（OPID/TAERRCODE/MDATA） | [4.9](../Chapter4_RFIDDataReply.md) |
| `ErrorReported` | 0xD0/0xD1 | `DeviceError`（OPERRCODE / SYSERRCODE + 中文） | [4.10/4.11](../Chapter4_RFIDDataReply.md) |
| `EventReported` | 0xE0/0xE1 | `DeviceEvent`（天線 Begin/End、LBT 干擾狀態） | [第 7 章](../Chapter7_EventNotification.md) |

> `TagReport` 需能容納不同盤點回包格式的欄位差異（基本 EPC、含 TID、含相位/頻率、自訂格式），
> 以可選欄位（nullable）表示「此格式沒有的欄位」。

## 4. 盤點觸發三模式（用 API 組出來）★工具核心驗證目標

對應使用者最關心的需求——必須能實測下列三種收集方式：

| 模式 | API 組合 | 預期行為（待實機驗證） |
|---|---|---|
| **單一觸發收集** | `SetOperationMode(非連續)` → `StartInventory()` | 盤一輪 → `TagReported` 串流數筆 → `InventoryDone`(0xC7) → 自動停止 |
| **開啟連續收集** | `SetOperationMode(連續)` → `StartInventory()` | 持續 `TagReported` 串流，不自動停 |
| **關閉連續收集** | `Cancel()` | 停止串流，回受理 STATUS |

**指令層應提供一個聚合方法／狀態機**封裝上述組合，例如：

```csharp
Task SingleInventoryAsync();        // 設非連續 + Start，收到 InventoryDone 視為完成
Task StartContinuousAsync();        // 設連續 + Start
Task StopContinuousAsync();         // Cancel
// 內部維護「目前是否盤點中」狀態，避免重複 Start / 在未啟動時 Cancel
```

> 規格行為以實機實測為準——若實機表現與上表不同（例如連續模式仍會自動停、或非連續也需手動停），
> 工具須能如實呈現差異，並在 [04 確認測試](04_確認測試規格.md) 記錄實測結論。這正是本工具的價值所在。
