## 1. 目的

開發一個在 PC（Windows）上執行的 **Console 互動工具**，用來在「拿到 RM300P 設備之後」：

1. **驗證序列通訊鏈路**：能開啟 COM port、與設備一來一回。
2. **驗證封包協定實作**：封包組裝、CHECKSUM、分包與解析都正確。
3. **實測標籤盤點的觸發行為**：親自確認設備是「下指令讀一次」還是「連續串流」，
   以及單次／連續開／連續關三種收集方式的真實表現（這是工具存在的核心理由）。
4. **作為後續基礎**：核心庫（與 UI 無關）可重用於階段 3 實作與未來系統整合；
   互動工具的測試流程可固化為階段 4/5 的驗收依據。

> 原因：不可逆或可能使設備/標籤損毀。文件已保留，工具第一版**不實作**，僅在規格保留位置與防呆說明（見 [05 安全](05_安全非功能與驗收.md)）。

## 4. 整體架構（四層）

工具分為**與 UI 無關的核心庫**與**Console 前端**兩大部分，核心庫內再分三層：

```
┌─────────────────────────────────────────────┐
│  UI 層（Console 前端）                         │  ← 光棒選單、即時盤點檢視、hex 監看
│  專案：RM300P.ConsoleTool                      │
├─────────────────────────────────────────────┤
│  指令層（Command Layer）                       │  ← 每個指令包成方法/模型；盤點事件回拋
│  ReadFirmwareVersion() / StartInventory() …    │     規格見 02
├─────────────────────────────────────────────┤
│  協定層（Protocol Layer）                      │  ← 封包編碼/解碼、CHECKSUM、STATUS/錯誤
│  PacketCodec / Checksum / ResponseParser       │     規格見 01
├─────────────────────────────────────────────┤
│  傳輸層（Transport Layer）                     │  ← COM port 開關、收發、STX/ETX 分包
│  ISerialTransport / PacketFramer               │     規格見 01
└─────────────────────────────────────────────┘
                      │
                 RM300P 設備（UART / 序列）
```

**分層原則**：
- 上層只依賴下層的介面，不跨層。
- **傳輸層與協定層完全不認識 UI**，可被 Console 工具、未來 GUI、自動測試共用。
- 盤點等「非同步串流」資料，由協定/指令層以**事件（event / callback）**往上拋，UI 訂閱後更新畫面。

## 5. 建議專案結構

```
（方案 RM300P_Tools.sln）
├─ src/
│  ├─ RM300P.Core/            ← 類別庫（傳輸+協定+指令三層）
│  │   ├─ Transport/          （ISerialTransport, SerialPortTransport, PacketFramer）
│  │   ├─ Protocol/           （Packet, Checksum, PacketCodec, ResponseParser, StatusCode）
│  │   ├─ Commands/           （各指令的 request/response 模型與 API）
│  │   └─ Models/             （TagReport, DeviceInfo, AntennaSetting … 等資料模型）
│  └─ RM300P.ConsoleTool/     ← Console 前端（光棒選單、畫面、hex 監看）
│      ├─ Menu/               （選單樹、光棒導覽元件）
│      ├─ Views/              （盤點即時檢視、設定表單、hex 監看）
│      └─ Program.cs
└─ tests/
   └─ RM300P.Core.Tests/      ← 單元測試（CHECKSUM、分包、編解碼可離線測）
```

> 此結構為建議；最終命名與專案切分於階段 3 實作時確認，但「核心庫與 UI 分離」為硬性原則。
