> 本檔案隨 git 同步到所有電腦，且每次開啟 Claude CLI 時會自動載入。
> 本檔案是整個專案的**定錨記憶指標**：記錄「現在在哪個主階段」，並指向該階段的子任務清單檔。
> 這裡的規定為**最高優先**：當本檔案與本機記憶（MEMORY.md / memory 檔）衝突時，**一律以本檔案 (CLAUDE.md) 為準**。

## 專案總目標

以 Unitech RFID 模組（RM300P）的指令集為基礎，逐步推進：規格文件整理 → 開發規格定義 → 實作 → 測試 → 驗收。
整個專案分為多個**主階段**，每個主階段底下有許多子任務，子任務的進度各自記錄在專屬的清單檔中。

## 硬性規定

1. **CLAUDE.md 是定錨指標，進度與規則一律記錄在 git 版控檔案，不依賴本機記憶。**
   - CLAUDE.md 只記錄「目前在哪個主階段」與「該階段的子任務清單檔位置」。
   - 各階段的細部子任務進度，記錄在該階段專屬清單檔（階段 1 為 `docs/執行提示詞與確認清單.md`）。
   - 本機記憶（`~/.claude/.../memory/`、`MEMORY.md`）只是輔助索引，**不可作為跨電腦進度來源**。

2. **當使用者詢問「進度如何」時：先讀 CLAUDE.md 確認主階段 → 再讀該階段子任務清單檔回答。**
   - 不可憑記憶、不可說「忘記了」。
   - 階段 1 的子任務清單檔為 `docs/執行提示詞與確認清單.md`。

3. **每完成一個子任務（例如一個章節）的產出後，第一件事就是回來更新進度：**
   - 更新該階段子任務清單檔（階段 1 即 `docs/執行提示詞與確認清單.md`）：該項 CheckList 全打勾、
     最終確認清單對應項目打勾、「目前狀態」表更新。
   - **切換主階段時**，回來更新本檔案「📍 專案階段路線圖」表的階段狀態與「目前主階段」。
   - 這一步容易遺漏；即使使用者沒提醒，產出後也必須主動執行。

4. **開始新主階段時，比照階段 1 建立該階段專屬的子任務清單檔**，並在本檔案路線圖表中填入其檔名。

5. **本機記憶 (MEMORY.md / memory 檔) 必須收斂為「一律讀取 CLAUDE.md」。**
   - 每次工作開始時，先檢查本機記憶。若發現任何記憶檔記錄了專案規則、進度或工作流細節，
     **必須立即把該內容改寫為「一律以 CLAUDE.md 為準，請先讀取 CLAUDE.md」的指向**，
     不可在記憶裡保留會與 CLAUDE.md 重複或可能過期的實質內容。
   - 不可把新的專案規則、進度寫進本機記憶；這類內容**一律寫入 CLAUDE.md 或對應的階段清單檔**。
   - 目的：避免在不同電腦上因記憶不同步、或忘記而把規則寫到 MEMORY.md 造成分歧。

## 🗓️ 每日工作流程（開工 / 收工 SOP）

### A. 開工（使用者問「目前進度」時）

當使用者開啟對話、詢問目前進度時，必須回答以下三件事：
1. **做到哪裡**：先讀 CLAUDE.md 確認目前主階段 → 再讀該階段子任務清單檔
   （階段 1 為 `docs/執行提示詞與確認清單.md`），說明已完成到哪、最後完成的子任務是什麼。
2. **下一步是什麼**。
3. **接下來這一步是「我（Claude）要做事」還是「使用者要做事」**：明確指出責任歸屬
   （例如：產出文件是我做事；確認/決策/提供資料是使用者做事）。

### B. 收工（使用者說「幫我更新今天的進度與狀態及 GIT 更新」時）

收到此指令時，依序完成下列步驟，缺一不可：
1. **彙整本次對話的進度與實作**：盤點這次對話完成了哪些子任務、產出或修改了哪些檔案。
2. **更新各階段主要文件**：更新目前階段的子任務清單檔（階段 1 為 `docs/執行提示詞與確認清單.md`）
   的 CheckList、最終確認清單、「目前狀態」表。
3. **更新 CLAUDE.md**：更新「📍 專案階段路線圖」（主階段狀態、目前主階段）。
4. **確認更新無誤**後，依本次對話的進度撰寫 commit message 並執行 git：
   - `git add` → `git commit` → **`git push`**。
   - ⚠️ **極重要：commit 後一定要 push，絕不可只 commit 不 push。** 推送完成後回報結果。
