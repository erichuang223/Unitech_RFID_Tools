# 2026-06-13 00:11 — 階段 2 啟動：工具方向三項拍板

## 使用者決策（原意鎖定）

針對「拿到 RM300P 後要開發什麼工具、做什麼確認測試」的討論，使用者拍板三項方向：

1. **技術棧 = C# / .NET**
   - 理由：在 Windows 上 `System.IO.Ports.SerialPort` 最直接，GUI 生態成熟，最適合序列設備工具。
2. **第一個工具形態 = 互動式 bench tool（優先）**
   - 讓人手動下指令、看即時盤點與原始 hex 收發；設備剛到貨時最快摸熟、最快驗證協定。
   - 自動化驗收 harness、純 SDK 整合留待之後。
3. **階段 2 先產出「開發規格文件」**
   - 依階段 2 定位（開發規格文件定義），先寫規格再進階段 3 實作。

## 我的理解與後續動作

- 階段 2 正式啟動，產出物 = 一份開發規格文件（定義 bench tool 的範圍、架構、傳輸/協定/指令三層、確認測試矩陣、安全規範、驗收標準）。
- 工具骨架共識：傳輸層（COM port + 分包）／協定層（封包編解碼 + CHECKSUM）／指令層（第 2~8 章指令 API），UI 疊在最上層。
- 確認測試採風險由低到高分階段：0 連線 → 1 握手(讀版本驗框架/校驗) → 2 唯讀資訊巡檢 → 3 設定 Get/Set 往返 → 4 盤點/讀寫 → 5 例外/破壞性(Lock/Kill) → 6 效能/穩定 → 7 韌體更新。

## 待我確認後才動工的剩餘決策

- GUI 框架：WinForms（快、適合工具）vs WPF（介面精緻）。
- 目標 .NET 版本：建議 .NET 8。
- 破壞性操作（Kill / 韌體更新）是否納入第一版工具範圍。
