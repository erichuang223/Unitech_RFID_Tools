## 專案架構

- 目前在階段 1（規格文件轉換）。

## 定錨與記憶規則

- `CLAUDE.md` 為專案定錨指標，記錄「目前在哪個主階段」與指向各階段子任務清單檔。
- 規則與進度一律放 git 版控檔案，因為工作目錄會 push 到 git 在另一台電腦運行，
  不能依賴單機的本機記憶。
- 本機記憶（MEMORY.md / memory 檔）必須收斂為「一律讀取 CLAUDE.md」，不得保留實質規則/進度。
- 衝突時一律以 CLAUDE.md 為準。

## 每日 SOP

- 開工：使用者問「目前進度」→ 我回答三件事：做到哪裡、下一步、這一步由誰做事。
- 收工：使用者說「幫我更新今天的進度與狀態及 GIT 更新」→ 彙整進度 → 更新階段清單檔
  → 更新 CLAUDE.md 路線圖 → git add/commit/**push**（commit 後一定要 push）。

## 規定 6（設計決策記錄方式）

- 使用者明確說明詳細規格/決策時，必須把使用者說的「全部」記到
  `.claude/daily/[yyyy-MM-dd]/[hh-mm].md`，因為我消化轉譯後會收斂、漏掉要點，
  所以原始說明要當下完整鎖住。
- 若內容是明確的操作流程／動作規範，要把該指示轉換對應寫進對應規格文件
  （例：輸出標題規範 → `docs/執行提示詞與確認清單.md`），用我看得懂、做得出來的方式重述。
- 會長期改變我行為的（SOP）另外同步 CLAUDE.md。
- 記完複述「我的理解是…，已記錄到 .claude/daily/...」讓使用者確認；分不清是否拍板就先問。
